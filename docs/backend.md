# Supabase Backend Configuration & Architecture Guide (Validated)

This document provides the complete, production-validated SQL schema, dual-project architecture, and configuration required for the **CaroHans Ventures ERMS** backend.

---

## 1. Dual-Project Architecture

To ensure operational isolation, security, and data integrity, the system utilizes two independent Supabase projects.

### 1.1 Primary ERMS Project (`yyelfynriosztsellphi`)
*   **Role:** The core business engine.
*   **Data:** Handles all production data (Inventory, Clients, Orders, Authentication, Financials).
*   **Performance:** Optimized for low-latency queries and real-time updates for the Admin Dashboard and Client Portal.
*   **Connection Policy:** Production traffic MUST use the **Transaction Pooler (Port 6543)** to prevent connection exhaustion.

### 1.2 Bug Reporting & Monitoring Project (`tsdbcatbcixdokdhxhxj`)
*   **Role:** Dedicated monitoring and debugging repository.
*   **Data:** Stores application error logs, bug reports, and user feedback.
*   **Isolation:** Prevents high-volume error logging from impacting the performance of the primary business database.
*   **Core Schema (Immutable):**
    *   **`public.bug_reports`**: `id`, `app_id`, `email`, `first_name`, `last_name`, `description`, `severity`, `status`.
    *   **`public.contacts`**: `id`, `email`, `phone`, `message`, `type`, `status`, `first_name`, `last_name`, `source`, `app_id`.

## 3. Database Schema (Primary Project)

### 3.1 Profiles Table
```sql
CREATE TABLE public.profiles (
    id uuid NOT NULL PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,
    role text NOT NULL CHECK (role = ANY (ARRAY['admin'::text, 'client'::text])),
    username text,
    email text,
    first_name text,
    last_name text,
    phone text,
    created_at timestamptz DEFAULT now(),
    updated_at timestamptz DEFAULT now()
);
```

### 3.2 Clients Table
```sql
CREATE TABLE public.clients (
    id integer GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    user_id uuid REFERENCES auth.users(id) ON DELETE SET NULL,
    first_name text,
    last_name text,
    username text UNIQUE,
    email text,
    phone text,
    address text,
    image text DEFAULT 'icon:User',
    color text DEFAULT 'text-primary',
    total_orders integer DEFAULT 0,
    total_spent numeric DEFAULT 0.00,
    last_order timestamptz,
    created_at timestamptz DEFAULT now()
);
```

### 3.3 Inventory Table
```sql
CREATE TABLE public.inventory (
    id integer GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    name text NOT NULL,
    category text NOT NULL,
    stock integer NOT NULL DEFAULT 0 CHECK (stock >= 0),
    price numeric NOT NULL DEFAULT 0.00,
    replacement_cost numeric NOT NULL DEFAULT 0.00,
    maintenance integer DEFAULT 0,
    image text,
    color text,
    sort_order numeric,
    created_at timestamptz DEFAULT now()
);
```

### 3.4 Orders Table
```sql
CREATE TABLE public.orders (
    id integer GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    client_id integer REFERENCES public.clients(id) ON DELETE SET NULL,
    client_name text,
    email text,
    phone text,
    status text NOT NULL DEFAULT 'Pending'::text,
    start_date date NOT NULL,
    end_date date NOT NULL,
    total_amount numeric NOT NULL DEFAULT 0.00,
    amount_paid numeric DEFAULT 0.00,
    penalty_amount numeric DEFAULT 0.00,
    deposit_paid boolean DEFAULT false,
    return_status text,
    item_integrity text,
    closed_at timestamptz,
    discount_name text,
    discount_type public.discount_value_type,
    discount_value numeric DEFAULT 0,
    created_at timestamptz DEFAULT now()
);
```

## 4. Business Logic (Functions & Triggers)

### 4.1 Security & Isolation
All critical RPCs and triggers use `SECURITY DEFINER` with a fixed `search_path` to prevent path manipulation attacks.
*   **`public.is_admin()`**: Validates if the current user has administrative privileges.

### 4.2 Automated Financials
The system uses Postgres triggers to ensure the `total_amount` column is always correct.
*   **Logic:** Whenever an item is changed or order dates/discounts are updated, the `calculate_expected_order_total` function runs.
*   **Guardrails:** Fixed discounts are hard-capped at the subtotal to prevent negative billing.

### 4.3 User Onboarding
*   **`handle_new_user()`**: Linked to the `on_auth_user_created` trigger. Automates profile and client record creation during signup, mapping metadata fields to respective columns.

### 4.4 Inventory & Stock Logic
*   **`submit_order()`**: Atomic RPC that performs a blocking availability check during the transaction to prevent overbooking.
*   **`get_available_stock()`**: Calculates real-time availability based on existing orders for selected dates.

## 5. Frontend-Backend Bridge

### 5.1 Image Resolution (`getImageUrl`)
Relative database paths (e.g., `chairs/gold.png`) are resolved to public CDN URLs:
`https://yyelfynriosztsellphi.supabase.co/storage/v1/object/public/assets/`

### 5.2 Caching Strategy
*   **Inventory Catalog:** Cached via `unstable_cache` with a **1 Second TTL** (`revalidate: 1`).
*   **Goal:** Instant user experience with near-real-time stock visibility.

## 6. Security & Performance

### 6.1 RLS Policies
Policies are optimized to use `InitPlan` sub-selects for `auth.uid()` to ensure O(1) evaluation.
*   **inventory**: Public Read; Admin Manage.
*   **orders**: Owner Read (lookup via email match to `auth.jwt()`); Admin Manage.

### 6.2 Indexing Strategy
*   **`idx_orders_client_id`**: Accelerates CRM order lookups.
*   **`idx_orders_date_range`**: Essential for real-time availability checks.
*   **`profiles_username_idx`**: Faster identity resolution during login.
